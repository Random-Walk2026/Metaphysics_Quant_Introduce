- 规则模块彼此独立，可单独维护和演进

- 功能扩展时更容易新增模块或新增分析层
- UI、数据、规则、AI 各层职责相对清晰

## 项目简介

MetaphysicsQuant 是一个把中国传统术数做成软件系统的长期工程项目。

它不是简单的“排盘工具”或“玄学脚本合集”，而是把传统命理、历法、卦象、择日、姓名学、数理系统中大量分散在古籍、口诀、经验规则里的知识，拆成了：

- 可执行的规则引擎
- 可维护的数据结构与 JSON 条文库
- 可复用的历法底层
- 可保存、检索、导出、回填的案例系统
- 可接入大模型的专题分析工作流
- 多模块统一的桌面端产品体验

如果用一句话概括，这个项目的核心价值是：

> **把传统术数中最难工程化的知识，做成一个可运行、可追溯、可扩展、可长期使用的软件平台。**

值得强调的是，项目并不把“算命准不准”当成卖点。它真正解决的是工程问题：**如何把非结构化的传统知识，转译成可复查、可版本化、可协作维护的结构化规则与数据。** 是否最终采纳某条断语，仍由使用者结合现实背景判断。

## 为什么这个项目难

这个项目的难点不在 UI，而在“知识系统的软件化”。

传统术数类系统常见的问题是：

- 规则散落在古籍、表格、口诀和人工经验里，缺少统一表达
- 不同系统对时间、历法、流派口径高度敏感，容错空间很小
- 输出往往不是固定字段，而是长篇结构化文本、专题断语和上下文组合
- 同一用户会反复分析、比较、修订，需要案例留存与回填
- 如果叠加 AI，必须先有可靠的规则层，不能让模型凭空“发挥”

MetaphysicsQuant 处理的是这些真正复杂的部分：

- 先做规则系统，再做交互层
- 先做结构化输出，再做 AI 解读
- 先做可复查的报告链路，再做结果展示

举几个具体的“魔鬼细节”，它们决定了系统能不能真正可用：

1. **早晚子时**：23:00–24:59 这一小时，不同流派对日柱的处理不同。系统在历法底层统一暴露 `setSect(1)` 之类的开关，让上层模块显式选择口径，而不是默默写死。
2. **闰月**：农历闰月在内部以负月份表示（例如闰三月记作 `-3`），所有依赖月柱的推导都必须正确识别。
3. **节气定月**：月柱不是按农历月初一切换，而是按 24 节气交接时刻精确切换，差几个小时就可能换柱。
4. **专题分层拼接**：同一个“婚姻”专题，可能同时引用苏氏盲派、五行精纪、三命通会、玉照神应真经、梁派理论等多个来源，需要按来源分层组织，而不是硬塞成一段。
5. **个性化神煞叠加**：择日时除了通用凶神，还要根据求测者本人八字叠加“冲福主年命、冲福主日命、箭刃煞、回头贡煞”等个人化凶神。

这些细节单看都不难，但要在七大系统里保持一致、可维护、可扩展，就需要把它们沉淀成统一的底层能力和数据结构——这正是项目的工程价值所在。

### 四柱八字：方法级接口最完整的主干模块

四柱模块当前的封装核心是 `destiny_bazi/SZBZ_6BzMs.py`。这一层不是单一排盘函数，而是一套围绕“原局 - 大运 - 流年 - 流月 - 中批 - 联动排盘图”组织的输出接口。

#### 1. 主要文本输出接口

- 原局命书：`bzms_num()` / `bzms_gz()`
- 简断命书：`bzmss_num()`
- 大运命书：`dyms_num()` / `dyms_gz()`
- 流年命书：`lnms_num()` / `lnms_gz()`
- 流月命书：`lyms_num()` / `lyms_gz()`
- 从当前年份起的 10 年流年中批：`zpln_num()`
- 流年与大运交接分段、流月与大运交接分段
- 反查出生时间

这里的 `num` 和 `gz` 两套入口，分别对应：

- `num`：按公历 / 农历年月日时分输入，适合真实出生资料
- `gz`：按年柱、月柱、日柱、时柱手输，适合古籍案例、课堂推演、已知八字反查

#### 2. 桌面端动作与按钮语义

- 原局分析、原局简断、大运分析、流年分析、流月分析
- 统一结果弹窗
- 从八字出生数据直接联动紫微斗数排盘图

`SZBZ_Widget_Gz.py`（“四柱批命”页）则面向干支手输场景，对应原局、大运、流年、流月分析。

也就是说，四柱模块不是“一个按钮出所有结果”，而是按命理师实际工作流拆成多种分析层级。

#### 3. 真实产物与可验证样本

仓库当前可见的**脱敏样本**说明，这些接口已经落到真实产物，而不是停留在设计目标：

- 四柱八字 · 原局分析 · Markdown：约 `36,035` 字符
- 四柱八字 · 原局分析 · TXT（参考模板）：约 `30,142` 字符
- 四柱八字 · 流年分析 · TXT：约 `10,246` 字符
- 四柱八字 · 综合分析 · Markdown（多份）

> 所有样本均已脱敏，不指向任何真实个人；文件名中的姓名一律以“示例案例”替代。

这类样本的意义不只是“有导出文件”，而是说明系统能稳定输出长文本、专题化、可归档、可复查的结构化结果。

#### 4. 工程展示时建议强调的点

- 这是一个已经形成“多层级报告树”的术数模块
- 不是只算八字，而是把原局、大运、流年、流月、交接段、中批和图形联动都做了

### 紫微斗数：文本命书、十二宫信息与统一排盘图

紫微斗数模块的封装核心是 `destiny_astrology/ZWDS_6ZwMs.py`。它目前已经形成了“原局专题 + 十二宫总览 + 流年专题 + 图形化排盘图”的组合输出。

#### 1. 主要文本输出接口

- 原局专题命书：`zwms_num()`
- 原局十二宫信息：`zwg12ms_num()`
- 流年专题命书：`zwlnms_num()`
- 大限命书接口预留：`zwdxms_num()`

需要实话实说的一点是：`zwdxms_num()` 这个接口已经留在封装层里，但当前桌面端的“全部大限分析 / 全部流年分析”仍是占位说明。这反而是对外展示时一个很真实的工程信号：项目并不靠文案把所有能力都说满，而是哪些已经落地、哪些还在演进，会如实区分。

#### 2. 图形输出接口

紫微斗数模块的一个关键展示点，不只是文本，而是**统一排盘图对话框** `ZwdsUnifiedChartDialog`：

- 四层切换：原局 / 大限 / 流年 / 流月
- 悬浮中心控件：年份、月份、大限切换直接叠加在图盘中心
- 层叠定位：多个排盘图弹窗自动错位排列，不相互遮挡
- 支持图片保存：PNG / JPG
- 也可被四柱模块复用，从八字出生数据直接联动打开

这说明紫微不是“后台算完一堆字”，而是已经形成图形化产品形态。

#### 3. 数据层与断语层

紫微斗数模块之所以值得展示，不只是因为“能画盘”，而是因为它把断语做成了数据驱动：

- 主星 / 辅星 / 四化基础数据：`star_zx.json`、`star_fx.json`、`star_4h.json`
- 宫位和专题断语：`topic_*.json`（性格、财运、事业、婚姻、健康、田宅、六亲等）
- 四化飞星断语：`topic_4h.json`（793 条“A 宫化禄权科忌入 B 宫”，按 12 个专题分类）
- 组合星匹配引擎：`StarCombinationMatcher`

断语数据的组织方式大致如下（示意，非真实条文）：

```jsonc
{
  "ZWDS": {
    "cbg": {                       // 财帛宫
      "single_star": {
        "紫微": "……单星断语……",
        "火星m": "……男性专属断语……",   // star+m / star+f 区分性别
        "火星f": "……女性专属断语……"
      },
      "combination_star": [
        {
          "stars": ["紫微", "天府"],     // 无序集合，子集匹配
          "description": "……组合断语（优先于单星）……"
        }
      ]
    }
  }
}
```

匹配引擎的几个工程化要点：

1. **无序子集匹配**：`_match_stars()` 用集合论判断盘面星曜是否命中某条组合，不要求顺序。
2. **优先级排序**：多星组合自动排在单星之前，命中更具体的断语优先输出。
3. **性别路由**：自动识别 `star+m` / `star+f`，按求测者性别选择对应条文。
4. **连接词智能**：四化星之间用“且”，普通星之间用“、”，十二长生神煞用“坐”而非“落”。
5. **空值过滤**：自动剔除空描述与格式异常条目。

可以对外明确说：

> **紫微斗数模块已经不是纯 if-else 规则堆砌，而是把星曜资料、专题断语和图形排盘分成了可维护的多层结构。** 新增一条断语只需改 JSON，不需要改代码——这正是“数据驱动”相对“硬编码”的核心优势。

### 易经解卦 / 六爻：一卦多断、多路神数并联的分析模块

六爻模块的综合编排核心是 `divination_hexagram/Divination.py`（`LyMs`）。它的特点不是单一“解一卦”，而是把多种解读路径并联起来，形成一个多角度分析总报告。

#### 1. 主要文本输出接口

- 原局分析：`lypd_ly()`
- 流年分析：`lypd_ln()`
- 全盘综合分析：`lypd_all()`
- 单专题报告：`get_topic_report(topic)` / `lypd_topic(topic)`
- 全部专题分类解卦：`lypd_topic_all()`

#### 2. 整合的多路解读

`lypd_all()` 不是简单拼接几段固定文案，而是把卦爻排盘、周易卦辞、五本古籍专题（卜筮正宗 / 黄金策 / 火珠林 / 增删卜易 / 易隐）、小六壬、干支神数、诸葛神数、太极神数、牙牌神数整合成一份总报告。

#### 3. 六爻专题结构

`lypd_ly()` / `lypd_all()` 中已经能看到非常明确的专题拆分：性格外貌、学业、事业、财运、感情、六亲（父母 / 兄弟 / 子女）、健康、阳宅、阴宅。这和项目其他模块保持了相似的“专题式组织”风格，说明产品层形成了统一表达习惯。

#### 4. 桌面端动作

`LY_Widget.py` 对应原局信息分析、流年信息分析、全盘信息分析和统一结果弹窗；起卦输入支持三数起卦、玄机数起卦、玄卦输入、时间起卦、动爻输入。

#### 5. 当前可见产物样本（脱敏）

- 易经解卦 · 综合分析 · TXT：约 `6,751` 字符
- 易经解卦 · 一卦多断 · TXT：约 `3,143` 字符

这说明六爻模块至少已经形成综合分析导出、一卦多断导出和脱敏样例留档。

### 择日合婚：主题择日、万年历、阴历年历与合婚参考

择日模块的封装核心是 `date_selection/ZR_3Ms.py`。它比常见“黄历查询器”更像一个围绕事件主题组织的日期分析引擎。

#### 1. 主要文本输出接口

- 万年历输出
- 阴历全年信息
- 专题择日（按 topic key 筛选）
- 合婚参考
- 专题元数据列举

这套接口对应了四种不同输出模式：起推日到次年同月同日的连续型日历、按阴历年范围展开的年历、针对单一专题的日期筛选、针对两组八字的匹配参考。

#### 2. 规则组织方式

这个模块真正有价值的地方，不只是专题很多，而是底层已经形成：分组（`_RAW_TOPIC_GROUPS`）、前端展示顺序（`TOPIC_GROUPS`）、专题到规则的归一化映射（`TOPIC_RULES`）、专题 good / bad 神煞信号（`_TOPIC_SIGNALS`）、规则 dispatch 表（`_TOPIC_RULE_DISPATCH`）。

所以更准确的说法是：

> **择日模块把传统择日条文转换成了可分组、可筛选、可推算、可解释的规则系统。**

### 姓名号码 / 英文数理：中文姓名学、起名、测字与三体系 numerology

#### 3. 工程含量体现在哪里

## 共享交互与通用 UI 层

如果只看业务模块，很容易把这个项目理解成“很多术数代码拼在一起”。但真正把它变成产品的是共享 UI 层（`ui/`）。这一层在简历项目里非常值得讲，因为它说明项目不是“功能堆叠”，而是有统一交互骨架。

这里集中维护最核心的通用组件：

- `TranslationThread`：多语言翻译型操作
- `ResultDialog`：统一结果展示、复制、导出、AI 解读入口
- `get_case_save_input()`：保存案例前的分类和备注输入
- `CompactTitleGroupBox`：统一局部容器视觉风格

外部读者很容易忽略这一层，但对软件工程来说，它意味着：不同模块没有各写各的线程模型、没有各写各的结果弹窗，案例保存交互是统一的，UI 细节不散落在每个 widget 里。

### 2. `ai_analysis_dialog.py`

这个文件不是“大模型调用本身”，而是桌面端的 AI 综合分析弹窗，承接模型选择、分析进度、结果展示、导出入口。这说明 AI 并不是藏在一个后台函数里，而是已经被纳入产品交互层。

对应案例备注浮动窗，用于从案例管理加载案例后查看、修改、覆盖保存备注。这类能力虽然不“炫”，但非常体现长期使用场景，而不是一次性 demo。

### 4. `app_bootstrap.py` 与 `icons/`

负责图标设置、应用启动辅助和默认 fallback 图标逻辑；独立的图标目录说明项目在桌面端已经有可识别的视觉资源，而不是纯开发期窗口。

## 支撑层：复用核心而非背景板

对外展示时，很多人容易只看“术数功能”。但真正让这个项目变成软件平台的，是下面这几层支撑基础设施。

### `lunar_solar/`：几乎所有术数模块都共享的时间引擎

> **项目把时间、节气、干支这类高敏感基础能力收敛到了统一底层，而不是让每个模块各算各的。**

### `case_character/`：把“一次分析”变成“长期研究与归档”

案例模块值得单独强调，因为很多玄学程序做到最后都停在“算一次就结束”。这个项目不是。它当前已经支持：

- 系统分组、分类管理、关键字搜索

- JSON 导入导出、Excel 导入导出
- Markdown / TXT 统一导出

支持系统覆盖四柱八字、四柱批命、紫微斗数、易经解卦、择日合婚、姓名号码、铁板神数、邵子神数。从产品角度看，这让仓库从“计算器”升级成了“研究工作台”。

> 注意：案例数据库 `cases.db` 是本地用户数据文件，属于私有资产，不在公开展示仓库中。

### `ai/`：把规则报告接到现代 LLM 工作流

AI 层现在不是一个简单的 `call_openai()` 文件，而是已经拆出：

- `analysis/`：AIAnalyzer、专题分析线程、解析重试
- `providers/`：OpenAI-compatible、Claude、Gemini、Qwen 等供应商抽象
- `analysis_graph.py`：LangGraph 风格分析流程
- `prompts/` + `prompt_manager.py`：Prompt 规则与契约兼容层
- `topic_schema.py`：报告专题拆分规则
- `core/`：缓存、Token、运行时基础设施

对外展示时，可以很明确地说：

> **AI 在这个项目里不是替代规则系统，而是建立在规则报告之上的第二层分析能力。** 先由规则层生成可追溯的原始报告，再按专题（☆ 标记）拆分给模型做解释、归纳和现实建议。真实 API Key 走环境变量 / `.env`，不进仓库。

## 这个项目体现了哪些技术能力

如果把它作为简历项目，它能够较完整地展示以下能力。

项目里的 AI 不是“接一个 API 就结束”，而是有完整的中间层设计：供应商抽象、兼容 OpenAI-style 接口、Prompt 管理、专题拆分、分析流程与结果校验。这更接近真实产品里的 AI integration，而不是 demo。

### 5. 本地数据与工作流设计

## 设计原则

项目在长期演化中逐渐沉淀出几条贯穿所有模块的设计原则，它们也是这套代码能在七大系统间保持一致的根本原因。

1. **规则先行，AI 在后**：任何专题报告都先由确定性规则引擎生成可追溯的原始文本，AI 只在其上做解释与归纳，不允许模型凭空“发挥”出新结论。
2. **数据与逻辑分离**：断语、条文、语料尽量沉淀到 JSON / 语料库，代码只负责匹配、组装和编排，新增内容不改代码。
3. **历法收敛到单一底层**：时间、节气、干支、八字这类高敏感能力只在 `lunar_solar` 实现一次，所有上层模块复用，避免口径分裂。
4. **专题式输出**：所有模块的报告都按统一的 `☆` 主题 / `§` 子条目结构组织，既便于人阅读，也便于 AI 按专题拆分。
5. **交互骨架统一**：线程模型、结果弹窗、案例保存、备注、图标启动集中在 `ui/`，业务模块不重复造轮子。
6. **如实区分“已落地”与“演进中”**：封装层可以预留接口，但桌面端会明确标注哪些功能已完成、哪些仍是占位，不靠文案把能力说满。
7. **长期可用优先于一次性 demo**：案例可保存、可检索、可回填、可导出，系统定位是“研究工作台”而非“算一次的计算器”。
